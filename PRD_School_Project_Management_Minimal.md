## 1. บทสรุป

พัฒนาพื้นที่ทำงานกลางสำหรับบริหารโครงการของโรงเรียน ตั้งแต่โครงการได้รับอนุมัติ การมอบหมายงาน การติดตามกำหนดเวลา การใช้จ่าย การเก็บหลักฐาน จนถึงการสรุปผล โดยต่อยอดจากโมดูล `Project`, `ProjectApprove`, `ProjectFollowup`, จัดซื้อ และการเงินที่มีอยู่แล้วใน SFMIS

แนวทาง Minimal คือให้ครูและเจ้าหน้าที่ตอบคำถามสำคัญได้จากหน้าเดียว:

1. โครงการนี้ทำอะไร ใครรับผิดชอบ และต้องเสร็จเมื่อใด
2. ขณะนี้งานอยู่ขั้นไหน มีงานใดล่าช้าหรือติดขัด
3. ได้รับงบเท่าไร ใช้จริงเท่าไร และเหลือเท่าไร
4. มีหลักฐานและผลลัพธ์ครบหรือยัง
5. ผู้อำนวยการหรือหัวหน้าแผนรับทราบผลแล้วหรือไม่

## 2. ที่มาและปัญหา

SFMIS ปัจจุบันรองรับการสร้างโครงการ การอนุมัติ การจัดซื้อ/จัดจ้าง และรายงานติดตามผลแล้ว แต่ข้อมูลการดำเนินงานจริงยังกระจายอยู่หลายหน้า และยังไม่มีรายการงานย่อยที่ระบุผู้รับผิดชอบ วันครบกำหนด และหลักฐานอย่างเป็นระบบ

ปัญหาหลัก:

- เจ้าของโครงการเห็นสถานะอนุมัติ แต่ไม่เห็นภาพรวมการลงมือทำ
- การติดตามงานอาศัยการสอบถามผ่านเอกสารหรือแชต
- เปอร์เซ็นต์ความก้าวหน้ากรอกด้วยตนเองและตรวจสอบที่มาได้ยาก
- งบประมาณโครงการกับยอดใช้จริงไม่ได้แสดงร่วมกันในพื้นที่เดียว
- หลักฐานกิจกรรม รูปถ่าย และเอกสารอยู่หลายแห่ง
- ผู้อำนวยการและหัวหน้าแผนมองไม่เห็นโครงการเสี่ยงล่าช้าในภาพรวม

## 3. แนวคิดจาก OCA/project ที่นำมาปรับใช้

| แนวคิด OCA | การปรับใช้ในโรงเรียนแบบ Minimal |
|---|---|
| Project Status | แยกสถานะการดำเนินงานออกจากสถานะอนุมัติ |
| Project Department | ผูกโครงการกับฝ่ายงานของโรงเรียน |
| Project Role / Stakeholder | ใช้บทบาท เจ้าของโครงการ ผู้ร่วมงาน ผู้ตรวจรับทราบ |
| Project Task / Stage | งานย่อยแบบ Kanban 5 สถานะ |
| Project Budget | แสดงงบแผน ยอดใช้จริง และยอดคงเหลือจากข้อมูล SFMIS |
| Milestone Status | คำนวณความก้าวหน้าจากน้ำหนักงานที่เสร็จ ไม่ใช้ timesheet |
| Project Template | ระยะถัดไปให้คัดลอกโครงการ/ชุดงานมาตรฐาน |
| Timeline | MVP ใช้กำหนดเริ่ม-สิ้นสุดและรายการใกล้ครบกำหนดก่อน Gantt เต็มรูปแบบ |
| Completion Blocking | ปิดโครงการไม่ได้หากยังมีงานค้างหรือหลักฐานบังคับไม่ครบ |

สิ่งที่ไม่นำมาใช้ใน MVP: timesheet, การคิดชั่วโมงทำงาน, โครงการซ้อนหลายชั้น, stock ระดับงาน, sales, recurrence และการตั้งค่า workflow ที่ซับซ้อน

## 4. เป้าหมาย

- ให้เจ้าของโครงการบริหารงานประจำวันได้จากหน้าเดียว
- ลดการติดตามงานนอกระบบและลดการกรอกข้อมูลซ้ำ
- เชื่อมแผน งาน จัดซื้อ การเงิน และผลลัพธ์เข้าด้วยกัน
- ทำให้ความก้าวหน้าและยอดใช้จริงตรวจสอบย้อนกลับได้
- แจ้งโครงการล่าช้า งบใกล้เต็ม หรือหลักฐานไม่ครบได้เร็ว
- รองรับการบริหารแยกตามโรงเรียน (`sc_id`) และปี (`sy_id`/`budget_year`)

## 5. ขอบเขตที่ไม่รวม

- ไม่แทนระบบอนุมัติโครงการและจัดซื้อ/จัดจ้างเดิม
- ไม่สร้างระบบบัญชีหรือทะเบียนงบประมาณชุดใหม่
- ไม่รองรับบุคคลภายนอกเข้าสู่ระบบใน MVP
- ไม่รองรับแผนภูมิ Gantt และ dependency ซับซ้อนใน MVP
- ไม่อนุญาตแก้ยอดใช้จริงด้วยมือ หากยอดนั้นมาจากธุรกรรมการเงิน

## 6. ผู้ใช้งานและสิทธิ์

| ผู้ใช้ | สิทธิ์หลัก |
|---|---|
| เจ้าหน้าที่แผน / หัวหน้าแผน (ประเภท 3, 6) | สร้างโครงการ จัดทีม ดูทุกโครงการในโรงเรียน ตรวจความครบถ้วน และติดตามภาพรวม |
| เจ้าของโครงการ | แก้ข้อมูลดำเนินงาน สร้าง/มอบหมายงาน แนบหลักฐาน และส่งสรุปผล |
| ผู้ร่วมโครงการ | ดูโครงการ อัปเดตงานที่ได้รับมอบหมาย และแนบหลักฐาน |
| พัสดุ / การเงิน (ประเภท 4, 5, 7, 8) | ดูข้อมูลโครงการและปรับปรุงข้อมูลผ่าน workflow จัดซื้อ/การเงินเดิม |
| ผู้อำนวยการ / Admin โรงเรียน (ประเภท 2) | ดูทุกโครงการ รับทราบรายงาน ขอให้แก้ไข และดู dashboard |
| Super Admin (ประเภท 1) | จัดการและตรวจสอบข้ามโรงเรียนตามสิทธิ์เดิม |

สิทธิ์เจ้าของโครงการและผู้ร่วมโครงการต้องกำหนดเป็นรายโครงการ ไม่อิงเพียง `tb_admin.type`

## 7. User Journey หลัก

### 7.1 เริ่มโครงการ

1. เจ้าหน้าที่แผนสร้างโครงการหรือเปิดโครงการเดิมที่อนุมัติแล้ว
2. ระบุเจ้าของโครงการ ทีม วันที่เริ่ม วันที่สิ้นสุด และผลลัพธ์ที่คาดหวัง
3. เพิ่มงานย่อยพร้อมผู้รับผิดชอบ วันครบกำหนด และน้ำหนักงาน
4. เมื่ออนุมัติแล้ว เจ้าของโครงการกด “เริ่มดำเนินงาน”

### 7.2 ดำเนินงาน

1. ผู้รับผิดชอบเปิด “งานของฉัน”
2. เลื่อนงานจาก ยังไม่เริ่ม ไป กำลังทำ / ติดขัด / เสร็จแล้ว
3. เมื่อเสร็จงาน แนบหลักฐานหรือบันทึกผลสั้น ๆ
4. ระบบคำนวณความก้าวหน้าโครงการอัตโนมัติ
5. งานใกล้ครบกำหนด งานเกินกำหนด และงานติดขัดแสดงบน dashboard

### 7.3 ใช้งบและจัดซื้อ

1. หากงานต้องจัดซื้อ เจ้าของโครงการกด “สร้างรายการจัดซื้อ/จัดจ้าง”
2. ระบบส่งต่อไป workflow `ProjectApprove` และพัสดุเดิม
3. เมื่อมีการตั้งเบิก/จ่าย ระบบรวมยอดใช้จริงกลับมายังหน้าโครงการ
4. ผู้ใช้เห็น งบอนุมัติ ใช้จริง ผูกพัน และคงเหลือ โดยไม่กรอกซ้ำ

### 7.4 ปิดและสรุปโครงการ

1. เจ้าของโครงการตรวจงาน หลักฐาน และตัวชี้วัด
2. ระบบ prefill เปอร์เซ็นต์ความก้าวหน้าและยอดใช้จริงลงรายงานผล
3. เจ้าของโครงการส่งสรุปผล
4. หัวหน้าแผนหรือผู้อำนวยการรับทราบ หรือส่งกลับให้แก้ไข
5. เมื่อเงื่อนไขครบ ระบบเปลี่ยนสถานะเป็น “ปิดโครงการ”

## 8. ขอบเขต MVP

### 8.1 หน้ารวมโครงการ

- แสดงโครงการตามปี โรงเรียน ฝ่ายงาน สถานะ เจ้าของโครงการ
- ค้นหาจากรหัสหรือชื่อโครงการ
- แสดง progress, งบประมาณ, ยอดใช้จริง, วันสิ้นสุด และสถานะความเสี่ยง
- มุมมอง “ทุกโครงการ”, “โครงการของฉัน”, “ต้องติดตาม”, “ปิดแล้ว”
- เรียงโครงการเร่งด่วนและเกินกำหนดขึ้นก่อน

### 8.2 พื้นที่ทำงานโครงการ

ใช้หน้าเดียว แบ่งเป็นแท็บ:

1. **ภาพรวม** - วัตถุประสงค์ เจ้าของ ทีม ระยะเวลา สถานะ ตัวชี้วัด
2. **งาน** - Kanban และรายการงานย่อย
3. **งบประมาณ** - งบอนุมัติ ผูกพัน ใช้จริง คงเหลือ และรายการอ้างอิง
4. **หลักฐาน** - เอกสาร รูปถ่าย และลิงก์ที่เกี่ยวข้อง
5. **รายงานผล** - ประวัติรายงานรายไตรมาส/ปลายปีและการรับทราบ

### 8.3 งานย่อย

ข้อมูลขั้นต่ำของงาน:

- ชื่องานและรายละเอียดสั้น
- ผู้รับผิดชอบ 1 คน
- วันที่เริ่มและวันครบกำหนด
- สถานะ
- น้ำหนักงาน 1-100
- เหตุผลเมื่อตั้งเป็น “ติดขัด”
- ผลการดำเนินงานและหลักฐานแนบ

สถานะงาน: `ยังไม่เริ่ม` → `กำลังทำ` → `รอตรวจ` → `เสร็จแล้ว` และสถานะพิเศษ `ติดขัด` / `ยกเลิก`

### 8.4 ทีมโครงการ

- หนึ่งโครงการมีเจ้าของได้ 1 คน
- มีผู้ร่วมโครงการหลายคน
- มีผู้ตรวจ/ผู้รับทราบได้หลายคน
- ผู้รับผิดชอบงานต้องเป็นสมาชิกโครงการ หรือผู้มีสิทธิ์แผนระดับโรงเรียน
- เก็บชื่อบทบาทในโครงการแยกจาก role ระบบ

### 8.5 ความก้าวหน้า

สูตร MVP:

```text
progress = ผลรวม(weight ของงานที่เสร็จแล้ว) / ผลรวม(weight ของงานที่ไม่ยกเลิก) × 100
```

- หากไม่ระบุน้ำหนัก ระบบกระจายน้ำหนักเท่ากัน
- งานสถานะ “รอตรวจ” ยังไม่นับว่าเสร็จ
- ผู้ใช้ดูรายละเอียดได้ว่าเปอร์เซ็นต์มาจากงานใด
- รายงานผลสามารถบันทึกผลผลิต ผลลัพธ์ ความพึงพอใจ ปัญหา และแนวทางแก้ไขตาม `ProjectFollowup` เดิม

### 8.6 งบประมาณ

- `งบอนุมัติ` ใช้ `pln_project.proj_budget`
- `ยอดผูกพัน` รวมรายการจัดซื้อ/สัญญาที่ยังไม่จ่าย
- `ยอดใช้จริง` รวมรายการที่จ่ายแล้วจากโมดูลการเงิน
- `คงเหลือ` = งบอนุมัติ - ยอดใช้จริง - ยอดผูกพัน
- คลิกยอดเพื่อเปิดเอกสารต้นทางได้
- หากไม่พบธุรกรรม ระบบแสดง 0 ไม่เปิดให้กรอกยอดทับ
- เตือนเมื่อใช้/ผูกพันเกิน 80% และห้ามสร้างรายการที่ทำให้เกินงบ เว้นแต่มี workflow โอนงบที่อนุมัติแล้ว

### 8.7 หลักฐาน

- ใช้ระบบ `tb_attachment` เดิม โดยเพิ่มชนิดอ้างอิง `project` และ `project_task`
- รองรับ PDF, รูปภาพ และไฟล์สำนักงานตามกติกา attachment กลาง
- ระบุชื่อหลักฐาน หมวด วันที่ และผู้อัปโหลด
- งานที่กำหนดว่าต้องมีหลักฐานจะเปลี่ยนเป็น “เสร็จแล้ว” ไม่ได้จนกว่าจะแนบไฟล์
- หน้าโครงการรวมหลักฐานจากงาน รายงานผล จัดซื้อ ตรวจรับ และการเงินแบบอ่านอย่างเดียว

### 8.8 การแจ้งเตือน

- งานครบกำหนดภายใน 7 วัน
- งานเกินกำหนด
- งานติดขัดเกิน 3 วัน
- โครงการใกล้สิ้นสุดแต่ progress ต่ำกว่า 80%
- รายงานผลรอรับทราบ
- งบใช้/ผูกพันเกิน 80%

ใช้โมดูลแจ้งเตือน/กระดิ่งเดิมของ SFMIS และไม่ส่งอีเมลใน MVP

## 9. สถานะและกฎการเปลี่ยนสถานะ

### 9.1 สถานะอนุมัติเดิม

คง `proj_status` ไว้สำหรับ `รออนุมัติ / อนุมัติ / ไม่อนุมัติ` เท่านั้น ห้ามนำไปใช้แทนสถานะการดำเนินงาน

### 9.2 สถานะดำเนินงานใหม่

```text
ร่าง → พร้อมดำเนินงาน → กำลังดำเนินงาน → รอตรวจสรุป → ปิดโครงการ
                         ↘ ติดขัด
ทุกสถานะ → ยกเลิก (ตามสิทธิ์และต้องระบุเหตุผล)
```

กฎ:

- เริ่มดำเนินงานได้เมื่อโครงการอนุมัติแล้ว มีเจ้าของ และมีวันสิ้นสุด
- ส่งตรวจสรุปได้เมื่อไม่มีงาน “ยังไม่เริ่ม/กำลังทำ/ติดขัด/รอตรวจ”
- ปิดโครงการได้เมื่อ progress = 100%, หลักฐานบังคับครบ และรายงานปลายปีถูกส่งแล้ว
- การเปิดโครงการที่ปิดแล้วต้องเป็นหัวหน้าแผนหรือผู้อำนวยการ และบันทึกเหตุผลใน activity log

## 11. แบบจำลองข้อมูลที่เสนอ

### 11.1 ปรับ `pln_project`

เพิ่มฟิลด์:

- `owner_admin_id` - เจ้าของโครงการแบบอ้างอิงผู้ใช้
- `start_date`, `end_date`
- `execution_status` - สถานะดำเนินงาน
- `progress_percent` - ค่า cache ที่ระบบคำนวณใหม่เมื่อ task เปลี่ยน
- `expected_output` - ผลผลิตที่คาดหวัง
- `success_indicator` - ตัวชี้วัดความสำเร็จ
- `closed_date`, `closed_by`, `cancel_reason`

คง `proj_owner` เดิมไว้เพื่อ backward compatibility และใช้เป็นชื่อ snapshot ระหว่าง migration

### 11.2 ตาราง `pln_project_member`

- `member_id`, `project_id`, `admin_id`
- `project_role`: `owner | member | reviewer`
- `role_name`, `note`
- `sc_id`, `up_by`, `del`, audit dates
- unique active member ต่อ `project_id + admin_id + project_role`

### 11.3 ตาราง `pln_project_task`

- `task_id`, `project_id`, `task_no`, `title`, `detail`
- `assignee_admin_id`, `start_date`, `due_date`
- `status`, `weight`, `sort_order`
- `evidence_required`, `result_note`, `blocked_reason`
- `completed_date`, `completed_by`
- `sc_id`, `sy_id`, `up_by`, `del`, audit dates

ไม่สร้าง subtask และ dependency ใน MVP เพื่อลดความซับซ้อน

### 11.4 ตารางและโมดูลเดิมที่นำกลับมาใช้

- `pln_project` - ข้อมูลโครงการและงบอนุมัติ
- `parcel_order` / `ProjectApprove` - การขอซื้อ อนุมัติ และยอดผูกพัน
- โมดูล Invoice/Check - ยอดใช้จริง
- `pln_project_followup` - รายงานผลรายช่วงและการรับทราบ
- `tb_attachment` - หลักฐาน
- Activity Log / Work Alert - ประวัติและแจ้งเตือน

ข้อเสนอเชิงโครงสร้าง: สำหรับโครงการใหม่ ไม่ควรสร้าง `parcel_order` ทันทีเมื่อสร้างโครงการ ให้สร้างเมื่อผู้ใช้เลือก “สร้างรายการจัดซื้อ/จัดจ้าง” เท่านั้น ส่วนข้อมูลเดิมยังคงลิงก์และใช้งานได้ตามปกติ

## 12. API ที่เสนอ

| Method | Endpoint | หน้าที่ |
|---|---|---|
| GET | `/api/projects/:id/workspace` | โหลดภาพรวม งาน ทีม งบ และหลักฐาน |
| PATCH | `/api/projects/:id/execution` | แก้เจ้าของ ระยะเวลา ตัวชี้วัด และสถานะดำเนินงาน |
| GET/POST | `/api/projects/:id/members` | โหลด/เพิ่มสมาชิก |
| DELETE | `/api/projects/:id/members/:memberId` | นำสมาชิกออก |
| GET/POST | `/api/projects/:id/tasks` | โหลด/สร้างงาน |
| PATCH | `/api/projects/:id/tasks/:taskId` | แก้ไขหรือเปลี่ยนสถานะงาน |
| DELETE | `/api/projects/:id/tasks/:taskId` | soft delete งาน |
| GET | `/api/projects/:id/budget-summary` | งบอนุมัติ ผูกพัน ใช้จริง คงเหลือ |
| GET | `/api/projects/:id/evidence` | รวมหลักฐานจากทุกโมดูล |
| POST | `/api/projects/:id/close` | ตรวจเงื่อนไขและปิดโครงการ |
| GET | `/api/projects/dashboard/:scId/:budgetYear` | dashboard โครงการของโรงเรียน |

ทุก endpoint ต้องตรวจ tenant จาก JWT และไม่เชื่อ `sc_id` จาก request เพียงอย่างเดียว

## 13. หน้าจอที่เสนอ

### 13.1 `/sfmis/plan-menu/projects`

- สรุปจำนวน: ทั้งหมด, กำลังทำ, ล่าช้า, ติดขัด, รอรับทราบ, ปิดแล้ว
- ตาราง/การ์ดโครงการพร้อม progress bar และ budget bar
- filter ปี ฝ่าย สถานะ เจ้าของ และความเสี่ยง

### 13.2 `/sfmis/plan-menu/projects/[id]`

- Header: รหัส ชื่อ สถานะ เจ้าของ วันสิ้นสุด ปุ่มดำเนินการหลัก
- Summary cards: progress, งานค้าง, งบคงเหลือ, หลักฐานครบ/ไม่ครบ
- Tabs: ภาพรวม งาน งบประมาณ หลักฐาน รายงานผล
- Kanban ต้องรองรับมือถือและมี List view สำรอง

### 13.3 `/sfmis/my-tasks`

- งานที่ฉันรับผิดชอบ เรียงตามเกินกำหนด/ใกล้ครบกำหนด
- เปลี่ยนสถานะ บันทึกผล และแนบหลักฐานได้โดยไม่ต้องเปิดหลายหน้า

## 14. Non-Functional Requirements

- หน้า workspace โหลดข้อมูลหลักภายใน 2 วินาทีที่ 100 โครงการ/โรงเรียน
- รองรับอย่างน้อย 500 งานต่อปีต่อโรงเรียน
- ทุก query ต้องกรอง `sc_id`, ปี และ `del = 0`
- ใช้ soft delete และ audit fields ตามมาตรฐานระบบ
- การเปลี่ยนสถานะ การปิด/เปิดใหม่ และการแก้งบต้องมี activity log
- UI ภาษาไทย รองรับ desktop, tablet และมือถือ
- วันที่ API/DB ใช้ CE (`YYYY-MM-DD`) และแสดงผลเป็น พ.ศ. ด้วย utility เดิม
- การเปลี่ยน schema ต้องมี TypeORM migration ก่อน production
- ห้ามเปิดเผยไฟล์หลักฐานข้ามโรงเรียนหรือแก่ผู้ไม่มีสิทธิ์ในโครงการ

## 15. Dashboard และตัวชี้วัด

- จำนวนโครงการตามสถานะและฝ่ายงาน
- โครงการเกินกำหนด / ติดขัด / ไม่มีการอัปเดตเกิน 14 วัน
- งานครบกำหนดใน 7 วัน
- งบอนุมัติ ผูกพัน ใช้จริง และคงเหลือรวม
- โครงการที่ใช้/ผูกพันเกิน 80%
- อัตราปิดโครงการตรงเวลา
- โครงการที่หลักฐานหรือรายงานผลยังไม่ครบ

## 16. Acceptance Criteria ระดับ MVP

1. เจ้าหน้าที่แผนเปิดโครงการเดิมและกำหนดเจ้าของ ทีม ระยะเวลา และตัวชี้วัดได้
2. เจ้าของโครงการสร้างและมอบหมายงานอย่างน้อย 5 งาน แล้วติดตามผ่าน Kanban ได้
3. เมื่อปิดงาน ระบบคำนวณ progress ใหม่โดยไม่ต้องกรอกเปอร์เซ็นต์เอง
4. งานที่บังคับหลักฐานไม่สามารถปิดได้หากยังไม่มีไฟล์แนบ
5. ยอดงบในหน้าโครงการตรงกับเอกสารจัดซื้อ/การเงินต้นทาง และคลิกย้อนกลับได้
6. ผู้ใช้จากโรงเรียนอื่นไม่สามารถอ่านหรือแก้โครงการ งาน สมาชิก และหลักฐานได้
7. ระบบระบุงานเกินกำหนดและโครงการเสี่ยงบน dashboard ได้
8. รายงานผลดึง progress และยอดใช้จริงมาเป็นค่าเริ่มต้นได้
9. โครงการปิดไม่ได้หากมีงานค้าง หลักฐานไม่ครบ หรือยังไม่ส่งรายงานสรุป
10. ผู้อำนวยการรับทราบรายงานและดูภาพรวมโครงการทั้งโรงเรียนได้

## 17. แผนส่งมอบ

### Phase 1 - Minimal Core

- Project workspace, member, task, Kanban/List
- execution status และ progress อัตโนมัติ
- budget summary จากข้อมูลเดิม
- attachment และ follow-up integration
- dashboard และ work alerts

### Phase 2 - Productivity

- คัดลอกโครงการ/ชุดงานมาตรฐาน
- timeline อย่างง่าย
- export รายงานโครงการและสรุปสำหรับประชุม
- comment/mention และแจ้งเตือนเพิ่มเติม

### Phase 3 - Advanced (พิจารณาภายหลัง)

- milestone หลายช่วง
- dependency ระหว่างงาน
- external stakeholder portal
- portfolio เปรียบเทียบหลายโรงเรียนสำหรับเขตพื้นที่

## 18. ความเสี่ยงและแนวทางลดความเสี่ยง

| ความเสี่ยง | แนวทาง |
|---|---|
| ข้อมูลซ้ำกับจัดซื้อ/การเงิน | ใช้ข้อมูลต้นทางและลิงก์อ้างอิง ไม่ให้กรอกยอดจริงซ้ำ |
| ผู้ใช้รู้สึกว่าต้องกรอกมากขึ้น | บังคับเฉพาะชื่อ ผู้รับผิดชอบ วันครบกำหนด สถานะ และหลักฐานที่จำเป็น |
| progress ไม่สะท้อนงานจริง | แสดงสูตรและรายการงานที่เป็นที่มาของเปอร์เซ็นต์ |
| role ระบบไม่ตรงกับทีมโครงการ | แยก project membership ออกจาก `tb_admin.type` |
| กระทบ workflow เดิม | คง endpoint/ตารางเดิม และเพิ่ม workspace เป็นชั้นเชื่อม |
| `parcel_order` ถูกสร้างโดยไม่จำเป็น | เปลี่ยนเป็นสร้างเมื่อเริ่มรายการจัดซื้อ พร้อมรองรับข้อมูลเก่า |

## 19. หลักการตัดสินใจผลิตภัณฑ์

- หนึ่งข้อมูลมีแหล่งจริงเพียงแห่งเดียว
- งานทุกชิ้นต้องมีเจ้าของและวันครบกำหนด
- ความก้าวหน้าต้องอธิบายที่มาได้
- หน้าเดียวต้องบอกได้ว่า “ใครทำอะไร เมื่อไร ใช้งบเท่าไร และมีหลักฐานครบหรือยัง”
- เริ่มจาก workflow คงที่ที่เข้าใจง่าย ก่อนเปิดให้ตั้งค่าเอง
- แนวคิดมาจาก OCA/project แต่พัฒนาให้เข้ากับ SFMIS โดยไม่คัดลอกโค้ด หากนำโค้ด AGPL มาใช้ต้องตรวจข้อกำหนดใบอนุญาตแยกต่างหาก
